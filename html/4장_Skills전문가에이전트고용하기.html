Chapter 04

스킬이란 무엇인가

스킬은 SKILL.md 파일 하나로 정의됩니다. 이 파일 안에 "어떤 상황에서 트리거되는지", "어떻게 행동해야 하는지", "결과물을 어떤 형식으로 낼지"가 담겨 있습니다. Claude는 스킬이 트리거되는 순간 이 파일을 읽고, 그 안의 역할대로 작동합니다.

SKILL.md 구조 — 스킬을 정의하는 파일
① 트리거 설명
언제 작동하는가
# 설명 (description)
사용자가 어떤 말을 할 때 이 스킬이 발동되어야 하는지 기술. Claude가 이 설명을 보고 스킬 목록에서 매칭합니다.
② 실행 지시
어떻게 행동하는가
# 지시 (instructions)
단계별 행동 지시, 사용할 도구, 참고할 파일, 판단 기준 등. 스킬의 핵심 로직이 담기는 곳입니다.
③ 출력 형식
무엇을 결과로 내는가
# 결과물 (output)
파일 생성, 채팅 응답, 외부 도구 호출 등 스킬이 완료됐을 때 사용자에게 돌아오는 것.
스킬 예시 — 각각이 하나의 전문가
frontend-design
구성 디자이너
design-system
디자인 시스템 설계자
pptx / docx
문서 제작자
aitoon-plot
웹툰 글작가
aitoon-ep
웹툰 그림작가
skill-creator
스킬 제작자
canvas-design
비주얼 디자이너
theme-factory
테마 디자이너

스킬은 Claude에게 "이 상황에선 이런 전문가로 행동해"라고 역할을 부여하는 것입니다. CLAUDE.md가 기본 성격을 설정한다면, 스킬은 필요할 때 꺼내 쓰는 전문가 카드입니다.

Chapter 04

스킬은 어떻게 실행되는가

스킬을 부른다고 해서 별도의 AI가 켜지는 게 아닙니다. 같은 Claude가 SKILL.md를 읽고 그 안의 지시에 따라 역할을 바꾸는 것입니다. 전체 흐름은 4단계입니다.

스킬 실행 흐름
1
트리거 감지
사용자 발화를 받은 Claude가 설치된 스킬 목록의 description과 비교합니다. "PPT 만들어줘" → pptx 스킬 매칭. /스킬명을 직접 입력하면 description 매칭 없이 즉시 호출됩니다.
💬 자동 감지: "HTML 파일로 만들어줘"
→ frontend-design 매칭
⌨ 직접 호출: /frontend-design
2
SKILL.md 로딩
Claude가 Read 툴로 해당 스킬의 SKILL.md를 읽습니다. 이 시점부터 Claude의 행동 방식이 스킬 지시를 따릅니다.
Read("/skills/frontend-design/SKILL.md")
3
스킬 지시에 따라 실행
SKILL.md 안의 단계별 지시를 따라 작업을 수행합니다. 디자인 철학 적용, 파일 생성, 필요 시 추가 스킬 호출까지.
4
결과물 전달
파일 생성 후 computer:// 링크로 전달, 또는 채팅 응답. 스킬이 정의한 출력 형식을 따릅니다.
하네스 5레이어에서의 위치
System Prompt기반층
MCP · 플러그인Layer 1 · Tools
CLAUDE.mdLayer 2 · Context
Skills ← 지금 여기Layer 3 · Context
스킬은 Layer 3에 위치합니다. 별도 AI가 켜지는 게 아니라, 같은 Claude가 SKILL.md를 읽고 같은 컨텍스트 안에서 역할을 전환하는 방식입니다. (서브에이전트는 p.21에서 별도 설명합니다.)

Chapter 04

런타임 합성 — 스킬이 스킬을 부른다

스킬을 하나씩 따로 쓰는 게 전부가 아닙니다. 메인 스킬이 실행되는 도중, 특정 부분에서 다른 스킬의 전문성이 필요하다고 판단하면 그 자리에서 호출해 활용할 수 있습니다. 메인 스킬이 주도권을 유지한 채 전문가를 불러 쓰고, 그 결과를 작업에 통합해 더 좋은 산출물을 만들어내는 방식 — 이것이 런타임 합성입니다.

실전 예시 — "내 디자인 시스템으로 랜딩 페이지 만들어줘"

메인은 끝까지 달린다
주도권은 처음부터 끝까지 메인 스킬이 쥡니다. 서브 스킬은 특정 파트에 잠깐 투입될 뿐, 작업을 넘겨받지 않습니다.
SKILL.md에 호출 지시 작성
어느 시점에 어떤 스킬을 부를지는 메인 스킬의 SKILL.md 안에 instructions으로 명시되어 있습니다.

런타임 합성의 핵심은 메인 스킬이 전문가를 필요한 순간에 호출해 쓴다는 점입니다. 혼자 모든 걸 처리하는 거대한 스킬 하나보다, 각자의 전문성을 가진 스킬들의 협업이 훨씬 정교한 결과를 만들어냅니다.

Chapter 04

스킬 생태계 — 설치하거나 직접 만들거나

스킬은 두 가지 방식으로 확장됩니다. 플러그인 마켓플레이스에서 설치하거나, 직접 SKILL.md를 작성해서 커스텀 스킬을 만드는 것입니다. 하나의 SKILL.md 파일만 있으면 어떤 작업이든 스킬로 만들 수 있습니다.

Chapter 04

함께 일하기 vs 통째로 맡기기

Cowork는 스킬 체인 방식뿐 아니라, 필요에 따라 서브에이전트를 호출해 작업을 위임하는 것도 가능합니다. 두 방식은 우열의 문제가 아닙니다. 나와 함께 진행하느냐, 통째로 맡기고 결과만 받느냐 — 작업의 성격에 따라 선택하는 것입니다.

스킬 체인
나와 함께 단계별로
같은 대화 안에서 스킬이 순서대로 역할을 전환하며 진행. 중간 결과를 내가 보고 방향을 조정할 수 있습니다.
잘 맞는 상황
· 순서가 있고 중간 검토가 필요한 작업
· 문서·디자인·코드 등 일반적인 제작
· 결과를 보면서 방향을 바꿔가며 진행
서브에이전트 (Subagent)
통째로 맡기고 결과만
별도 컨텍스트에서 독립적으로 실행. 작업 전체를 위임하고 완성된 결과만 받아봅니다.
잘 맞는 상황
· 대량의 파일을 병렬로 처리해야 할 때
· 완전히 독립적으로 실행해야 하는 작업
· 중간 과정 없이 결과물만 필요한 경우
구분 스킬 체인 서브에이전트
실행 공간 같은 대화 안 별도 컨텍스트
진행 방식 단계별 역할 전환 독립 실행 · 결과만 반환
중간 개입 가능 (단계마다 확인) 없음 (완료 후 수령)
병렬 처리 어려움 가능 (여러 에이전트 동시)
설정 난이도 낮음 (SKILL.md) 높음 (에이전트 설계 필요)

두 방식은 경쟁 관계가 아닙니다. 나와 함께 진행하며 방향을 조정해야 한다면 스킬 체인, 작업을 완전히 위임하고 결과만 받아도 된다면 서브에이전트 — 작업의 성격이 방식을 결정합니다.
